失败聚类

页面 · 成长 · 5 分钟

把相似的亏损交易归为一类，找出模式背后的共同原因，精准定位策略的薄弱环节。

TL;DR

失败聚类页面将你的亏损交易按特征自动分组，揭示隐藏的模式：比如"所有亏损都发生在低流动性时段"或"止损距离过近导致被频繁扫出"。页面分为三个标签页：失败聚类（展示聚类卡片，含交易数、总亏损、严重等级和建议修复）、Regime 矩阵（市场体制 x 失败类型的交叉热度表）、拒单原因（交易被系统拒绝的统计）。每个聚类卡片附带一条 AI 建议的修复方案，可直接生成影子策略。

谁用 · 何时用

所有希望系统性改进策略的交易者。当你从增长回顾中发现某策略持续亏损，或直觉上觉得亏损有共同特征时，打开失败聚类页面。也适合定期（例如每周）检查一次，确保没有新的失败模式悄悄形成。聚类数量越多说明亏损越分散，需要关注；如果聚类数量少但单个聚类亏损大，则针对性修复效果最好。

页面解剖

  1. ① 聚类卡片列表 — 每个聚类代表一类失败模式，包含标签、严重等级（严重/高/中/低）、交易数、总亏损和平均亏损百分比。底部显示亏损量条形图（最严重的聚类占满宽度）和 AI 建议的修复方案。点击"示例"标签可查看该聚类中的代表性交易 ID。
  2. ② Regime 矩阵 — 市场体制（趋势/震荡/高波动/恐慌等）与失败类型的交叉热度表。颜色越深表示亏损越大。快速判断：你的策略在哪种市场条件下最容易亏损？例如在"高波动"体制下频繁出现"止损被扫"问题，说明止损策略需要调整。
  3. ③ 拒单原因统计 — 按原因列出交易被系统拒绝的频次。常见原因包括：日亏损上限触发、快照过期、AI 缓存过期、持仓上限、波动率过高。高频拒单可能说明策略风控参数或基础设施配置需要调整。

关键指标怎么读

  • 聚类严重等级 — 基于亏损金额和交易数综合评定。严重（Critical）聚类需要立即关注；高（High）应在下次优化中处理；中（Medium）和低（Low）可监控但无需紧急行动。
  • 聚类亏损占比 — 每个聚类亏损占最大聚类亏损的比例（条形图宽度）。聚焦占比最高的 1-2 个聚类，修复效果最明显。
  • Regime 矩阵颜色深度 — 颜色越深表示该体制+失败类型的组合亏损越大。如果某个交叉格颜色显著深于其他，这就是优先修复方向。
  • 拒单频次 — 高频拒单可能说明策略过于激进或系统配置有问题。例如"日亏损上限"频繁触发说明风险预算不合理；"快照过期"频繁出现则需检查网络/Redis 稳定性。

常见操作 · 一步步

  1. 查看聚类详情 — 在"失败聚类"标签页中，滚动浏览各聚类卡片。关注严重等级为"高"或"严重"的聚类。阅读 AI 建议的修复方案，了解该模式产生的原因和可能的改进方向。
  2. 分析 Regime 矩阵 — 切换到"Regime 矩阵"标签页。查看哪些市场体制与哪些失败类型相关。例如，如果在"高波动 + 止损被扫"格中看到深色，说明策略在高波动市场中的止损设置需要放宽。
  3. 生成影子策略 — 在聚类卡片底部，点击"生成影子策略"（如果可用）。系统自动创建一个包含 DSL 补丁的影子策略草稿，用于修复该聚类对应的失败模式。影子策略需经过校验、回测、模拟盘等完整流程才能上线。
  4. 检查拒单原因 — 切换到"拒单原因"标签页。查看各原因的发生频次。如果某个原因频次异常高，考虑调整相应的风控参数或排查基础设施问题。

幕后

失败聚类由 failure_clustering 服务在每次增长回顾运行时触发。系统从 ExecutionLedger 读取所有亏损交易（profit_pct < 0），通过标签系统将相似亏损归入同一聚类。标签系统分析多个维度：交易时间（低流动性时段）、市场体制（高波动/震荡）、止损距离、持仓时间、AI 缓存状态等。聚类结果存入 failure_clusters 表，旧聚类自动归档。影子策略生成器（shadow_strategy_generator）读取活跃聚类，结合目标策略的最新 DSL 版本，由 DSLPatchService 生成 JSON Patch 补丁，最终创建 ShadowStrategyDraft 记录。整个流程无需人工干预，聚类结果实时更新。
