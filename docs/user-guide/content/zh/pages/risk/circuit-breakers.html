熔断器

页面 · 风控 · 5 分钟

熔断器是最后一道防线——当日亏损、连续亏损或波动率超出阈值时，系统自动停止所有交易，保护账户不被一次行情击穿。

TL;DR

熔断器页面以时间线形式展示所有历史熔断记录。顶部有筛选标签：全部、紧急停止、Kill Switch、亏损锁、手动平仓、安全模式。每条记录显示熔断类型、触发时间（相对时间）、reason codes、关联命令 ID 和对账 ID。如果从未触发过熔断，页面显示一个绿色脉冲盾牌徽标和"ALL CLEAR"状态。

谁用 · 何时用

熔断器页面主要用于事后复盘。当系统突然停止交易时，先看执行中心确认状态，然后来这里查看触发原因。每天收盘后也可以快速浏览，确认当天没有意外触发。如果你在运行新策略的初期阶段（实盘小资金），建议每隔几小时检查一次。

页面解剖

  1. ① 筛选标签 — 按熔断类型筛选：全部、紧急停止、Kill Switch、亏损锁（合并日/周）、手动平仓、安全模式。选中标签高亮对应颜色。
  2. ② 熔断记录（日亏损锁） — 左侧时间线圆点，颜色对应类型。卡片显示类型徽章、相对时间（如"2 小时前"）、reason codes 标签和关联命令 ID。
  3. ③ 熔断记录（紧急停止） — 红色表示手动紧急停止。附带关联的 CMD（命令）和 RECON（对账）ID 供追踪。
  4. ④ ALL CLEAR 状态 — 当没有任何熔断记录时，显示绿色脉冲盾牌徽标和 "ALL CLEAR" 文字。

关键指标怎么读

  • 熔断类型 — 紧急停止（手动触发，红色）、Kill Switch（自动保护，红色）、日/周亏损锁（达到亏损限额，橙红色）、手动平仓（黄色）、安全模式（紫色）。类型决定了触发后的恢复流程。
  • 触发时间 — 以相对时间显示（"刚刚"、"5 分钟前"、"2 小时前"、"3 天前"）。越近的触发越需要关注恢复状态。
  • Reason Codes — 具体触发原因，如 daily_loss_limit_reached（日亏损限额到达）、manual_trigger（手动触发）、consecutive_loss_limit（连亏超限）。
  • 关联 CMD / RECON ID — 当熔断由某个命令触发或与某次对账相关时，显示关联 ID 供事后审计追踪。

常见操作 · 一步步

  1. 查看所有熔断记录 — 打开熔断器页面，按时间线从上到下浏览。最新记录在最上面。
  2. 按类型筛选 — 点击顶部的筛选标签，只查看特定类型的熔断记录。再次点击同一标签取消筛选。
  3. 排查触发原因 — 展开记录查看 reason codes。常见原因：daily_loss_limit_reached（日亏损超限）、consecutive_loss_limit（连亏超限）、manual_trigger（手动触发）。
  4. 恢复交易 — 熔断器页面只提供记录查看。恢复交易请到风控中心解除紧急锁定，再到执行中心手动恢复各策略会话。

幕后

熔断器数据来自 /api/risk/circuit-breakers。后端从 PostgreSQL 的 circuit_breaker_events 表查询所有类型为 emergency_stop、daily_loss_lock、weekly_loss_lock、kill_switch、manual_force_close 的事件，按时间倒序排列。未解决的熔断事件（resolved = false）会将整体状态标记为 "tripped"；全部解决则为 "healthy"。数据库查询失败时回退到模拟数据。熔断事件的写入发生在后端其他服务的风控逻辑中（如每日交易循环中的熔断检查）。
