实盘准入检查

页面 · 总览 · 5 分钟

在启动实盘交易之前，系统会对基础设施、交易所连接、性能指标进行全面检查，告诉你"现在可以上线吗"。

TL;DR

实盘准入检查是一个安全门。它汇总所有关键系统（Fast Track、Redis、Freqtrade、交易所 API）的健康状态，计算一个 0-100 的准入分数，并明确告诉你当前允许的交易模式：仅模拟、小仓实盘、还是全量实盘。任何阻断项都会阻止你进入更高风险的模式。

谁用 · 何时用

每次从模拟交易切换到实盘之前，或者怀疑系统有问题时。新策略上线前也建议先过一遍准入检查，确认交易所连接正常、网络延迟在阈值以内。日常运营中如果看到 Dashboard 顶部状态栏出现异常，也应该来这里看看具体问题出在哪里。

页面解剖

  1. ① 准入分数环 — 环形进度条显示综合评分（0-100）。绿色 ≥80 分、黄色 50-79 分、红色 <50 分。分数由所有检查项的状态加权计算得出。
  2. ② 状态横幅 — 显示当前准入状态（LIVE_READY / LIVE_SMALL_READY / PAPER_ONLY / NOT_READY）。下方列出三种交易模式的可用性：Paper（模拟）、Live Small（小仓实盘）、Full Live（全量实盘），绿色勾表示允许，红色叉表示被阻挡。
  3. ③ 阻断项与警告项 — 红色背景的"阻断项"必须解决才能提升准入级别。黄色背景的"警告项"建议解决但不强制。每项显示错误代码和详细消息。
  4. ④ 系统检查网格 — 四列网格展示每个子系统的健康状态：Fast Track 延迟、Redis 往返时间、Freqtrade 运行状态、交易所 API 权重。每个格子包含状态圆点（绿/黄/红）、指标名称、当前值和阈值。
  5. ⑤ 操作栏 — "重新检查"按钮手动触发新的健康检查。"启动模拟"和"启动小仓实盘"按钮在对应模式可用时亮起。

关键指标怎么读

  • 准入分数 (0-100) — 最重要的全局指标。80 分以上允许全量实盘，50-79 分仅允许小仓实盘，低于 50 分只允许模拟交易。分数受最差的检查项影响最大——一个"failed"状态的检查项会大幅拉低总分。
  • Fast Track 延迟 — 系统内部通信的往返时间，理想值 <200ms。延迟过高说明基础设施可能存在瓶颈。
  • Redis RTT — Redis 缓存的响应时间，理想值 <50ms。如果 Redis 响应慢，会影响所有依赖缓存的数据读取。
  • Freqtrade 状态 — 核心交易引擎的在线状态，必须为 "running" 才能实盘。任何其他状态（如 "stopped"、"error"）都是阻断项。
  • 交易所 API 权重 — 交易所 API 的剩余配额百分比。低于 90% 会触发警告，低于阈值可能导致订单提交被限流。
  • 准入状态 — 四种状态：LIVE_READY（绿灯，全量可交易）、LIVE_SMALL_READY（橙灯，仅小仓）、PAPER_ONLY（黄灯，仅模拟）、NOT_READY（红灯，不可交易）。

常见操作 · 一步步

上线前检查

  1. 打开"实盘准入检查"页面，查看左上角的准入分数环是否达到绿色区域（≥80）。
  2. 检查"阻断项"区域是否有红色项目。如果有，记录错误代码并逐一排查解决。
  3. 检查"警告项"区域，判断是否可以接受。例如"交易所 API 权重剩余偏低"虽然不阻断，但在高交易频率场景下需要留意。
  4. 确认状态横幅中目标模式的勾选状态（例如确认 Full Live 是绿色勾）。
  5. 点击"重新检查"按钮触发一次新的健康检查，确认最近修复的问题已经解决。
  6. 点击对应模式的操作按钮（如"启动小仓实盘"）开始交易。

幕后

页面数据通过 /api/overview/live-readiness 端点获取，使用 APIOverview 服务。后端综合 Fast Track 心跳、Redis 连通性测试、Freqtrade 进程状态和交易所 API 余量计算评分和准入状态。点击"重新检查"按钮会调用同一个端点进行强制刷新。后端不可用时降级到模拟数据（Mock），模拟数据默认评分 86 分，状态为 LIVE_SMALL_READY。
