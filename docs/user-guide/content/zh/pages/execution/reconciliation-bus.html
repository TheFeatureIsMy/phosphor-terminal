对账总线

页面 · 执行 · 5 分钟

对账总线确保你看到的持仓数据与交易所的真实状态一致——任何偏差都会被捕获并标记为差异。

TL;DR

对账总线是 AlphaLoop 的"数据一致性警察"。它定期比较本地数据库中的持仓记录与 Freqtrade/交易所的真实状态，找出差异（如本地有但交易所无、交易所存在但本地缺失、数量不匹配），并在页面中以 Command Bus 时间线和对账运行记录的形式展示。页面顶部有刷新交易所状态的按钮，你可以手动触发一次完整的对账流程。

谁用 · 何时用

系统管理员或高级交易员在以下场景使用：怀疑订单状态不同步时（比如本地显示已成交但交易所显示挂单中）、运行回测/模拟后验证数据一致性、或者排查"幽灵持仓"问题时。一般用户在正常情况下不需要频繁访问此页面——对账是自动运行的。

页面解剖

  1. ① 标题区 — 显示页面标题和副标题 "Command Bus · State Lease · Exchange Finality"。右上角有"刷新交易所状态"按钮，手动触发一次完整对账。
  2. ② 对账状态 — 整体对账健康度：healthy（正常）/ reconciliating（对账中）/ degraded（有差异需关注）/ error（对账失败）。
  3. ③ Command Bus Timeline — 最近 20 条命令执行记录，每条显示命令类型、ID 和状态（completed / pending / failed / timeout）。命令包括 start_dryrun、place_order、stop_session 等。
  4. ④ 对账记录 — 最近 10 次对账运行，每次显示差异数量。0 discrepancies 为绿色（完全一致），大于 0 为橙色（需要调查）。黄色三角形表示附加的 reason_codes。
  5. ⑤ 状态色码 — 绿=完成/成功 / 黄=待处理/运行中 / 红=失败/错误 / 橙=超时。

关键指标怎么读

  • discrepancies（差异数） — 对账运行时发现的本地与交易所状态不一致的数量。0 是最好的情况。任何大于 0 的差异都应该被调查。常见的差异类型：local_only（本地有但交易所无）、remote_only（交易所存在但本地无记录）、amount_mismatch（数量对不上）。
  • 对账状态 — healthy 表示最近一次对账无差异且一切正常；reconciliating 表示系统正在运行对账；degraded 表示检测到差异但系统仍在运行；error 表示对账流程本身失败。
  • Command Bus 命令状态 — completed 表示命令已成功执行；pending/running 表示命令还在处理中；failed 表示命令执行出错；timeout 表示命令执行超时（通常意味着交易所响应过慢）。

常见操作 · 一步步

  1. 刷新交易所状态 — 点击右上角"刷新交易所状态"按钮 → 系统从 Freqtrade REST API 拉取最新持仓和订单数据 → 自动与本地记录比对 → 更新差异列表。
  2. 查看对账详情 — 点击对账记录行，可展开查看具体的差异明细（当前版本中差异行直接显示在页面中，详细 drill-down 在开发中）。
  3. 排查差异 — 当看到 discrepancies > 0 时，检查差异类型：local_only 通常意味着本地记录未同步到交易所（可能是网络问题）；remote_only 意味着交易所的订单未回写到本地（可能是回调丢失）；amount_mismatch 需要检查是否有部分成交未更新。

幕后

对账总线由 ReconciliationService（reconciliation_service.py）驱动。对账流程分三步：首先 start_reconciliation 创建对账事件并将 FreqtradeRun 标记为 "reconciliating"；然后 execute 对比本地 ExecutionPosition 与 Freqtrade REST API 返回的持仓数据，逐交易对检查存在性和数量一致性；最后 complete 标记对账完成并将 FreqtradeRun 恢复为 "running"。每次对账的每一步都通过 ExecutionLedgerEvent 写入账本（事件类型 PULSEDESK_RECONCILIATION_STARTED / COMPLETED）。后端 BFF 端点 /api/reconciliation/bus 和 /api/reconciliation/runs 提供前端数据，查询最近 20 条 Command Bus 命令和最近 50 条对账记录。
