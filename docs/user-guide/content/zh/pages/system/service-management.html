服务管理

页面 · 系统 · 5 分钟

一站式监控所有后端服务的健康状态——Freqtrade、Redis、PostgreSQL、AI Provider，并管理推理任务与隐私策略。

TL;DR

服务管理页面聚合了整个 AlphaLoop 系统的运行状态。它分为两大板块：上方是 AI 服务管理（Provider 总览、任务路由矩阵、模型运行状态、推理队列、隐私策略），下方通过 Dashboard 上的服务健康卡片快速反映 Freqtrade、Redis、PostgreSQL 等基础设施的连通性。当某个服务出现延迟或宕机时，颜色会从绿色变为黄色或红色，让你在交易前就能发现问题。

谁用 · 何时用

运维人员或在实盘前检查系统状态的交易者。每天早上启动 AlphaLoop 后，先看一眼服务管理确认所有依赖都在线。当 AI 信号生成异常慢或交易执行不响应时，也来这里排查是哪个环节出了问题——Freqtrade 断连、AI Provider 限流、还是模型 OOM。

页面解剖

  1. ① 服务状态网格 — 所有依赖服务的实时健康指示。绿点 = 正常响应，黄点 = 延迟升高，红点 = 服务不可用。旁边显示延迟毫秒数。
  2. ② AI Provider 总览 — 列出所有已配置的 AI 服务（Ollama、OpenAI、Anthropic 等），显示在线/离线状态、延迟、模型数和失败率。每张卡片有"测试"按钮可手动验证连通性。
  3. ③ 任务路由矩阵 — 定义每个 AI 任务类型（生成、分析、情绪、检索）的 Provider 路由策略。支持主/备路由和多种策略：failover、round-robin、local-only、cost-opt。
  4. ④ 推理队列 — 实时显示正在执行和排队的推理任务。每行显示模型名、任务类型、预估费用和状态。可取消排队中的任务。
  5. ⑤ 隐私策略矩阵 — 按数据类型标记是否允许本地推理和云端 API 调用。交易数据和密钥强制本地，研究数据可云端。

关键指标怎么读

  • 服务状态颜色 — 绿色 = 健康（延迟 < 100ms），黄色 = 降级（延迟 100ms-1s 或部分限流），红色 = 宕机（不可用）。如果 Freqtrade 变红，交易执行将中断。
  • Provider 失败率 — 反映 AI 服务调用失败的百分比。持续高于 5% 表示需要切换路由策略或检查 API Key 是否过期。
  • 推理队列积压 — 排队中的任务数量持续增长说明当前 Provider 吞吐不足，应考虑添加备用 Provider 或升级模型实例。
  • 路由策略 — failover（主路由失败自动切备用）、round-robin（轮询分摊负载）、local-only（仅本地推理）、cost-opt（优先选成本最低的可用 Provider）。

常见操作 · 一步步

  1. 打开服务管理页面，快速扫视顶部服务状态网格——确认所有绿灯亮起。
  2. 如某个服务变黄或变红，点击对应服务的"测试"按钮确认是否瞬时波动。
  3. 切换到 Provider 总览区域，检查各 AI Provider 的失败率和延迟。如 OpenAI 持续高延迟，考虑将任务路由切换到 Anthropic 备用。
  4. 在任务路由矩阵中调整策略——例如将生成任务从 failover 改为 cost-opt 以降低费用。
  5. 查看推理队列，如有长时间排队的任务可取消或提升优先级。
  6. 检查隐私策略矩阵，确保敏感数据（交易数据、密钥）仅允许本地推理。

幕后

服务状态数据来自后端 /api/system/status 和 /api/system/dependencies 端点。Freqtrade 状态通过 REST API 查询实时交易引擎状态；Redis 和 PostgreSQL 通过后端连接池探活。AI Provider 数据通过 APIAIProviders 服务并行获取——Provider 列表、模型运行状态、路由规则和隐私策略各走独立请求。推理队列由 APIInference 管理，任务取消通过 /api/inference/jobs/<id>/cancel 触发。所有数据刷新通过页面顶部的刷新按钮手动触发。
