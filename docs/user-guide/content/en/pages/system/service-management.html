Service Management

Page · System · 5 min

A single-pane view of every backend service's health — Freqtrade, Redis, PostgreSQL, AI Providers — plus inference queue management and privacy policy configuration.

TL;DR

Service Management aggregates the health of all AlphaLoop dependencies into one page. It has two main zones: AI service management (Provider overview, task routing matrix, model runtime status, inference queue, privacy policy) and an infrastructure health grid covering Freqtrade, Redis, PostgreSQL, and exchange APIs. Color-coded status dots let you spot problems at a glance before placing trades.

Who uses it & when

Operators checking system readiness before live trading, or anyone troubleshooting slow signal generation or failed order execution. Open it first thing after launching AlphaLoop to confirm every dependency is green. Also visit when AI responses stall — check whether the Provider is rate-limited, the model is OOM, or the inference queue is backlogged.

Page Anatomy

  1. ① Service status grid — Real-time health indicators for all dependencies. Green = responding normally, yellow = elevated latency, red = unavailable. Latency in milliseconds shown next to each dot.
  2. ② AI Provider overview — Every configured AI service (Ollama, OpenAI, Anthropic, etc.) with online/offline status, latency, model count, and failure rate. Each card has a "Test" button for manual connectivity verification.
  3. ③ Task routing matrix — Maps each AI task type (generation, analysis, sentiment, retrieval) to a Provider routing strategy. Supports primary/fallback with failover, round-robin, local-only, and cost-opt strategies.
  4. ④ Inference queue — Live view of running and queued inference jobs. Each row shows model name, task type, estimated cost, and status. Queued jobs can be cancelled.
  5. ⑤ Privacy policy matrix — Per data type, marks whether local inference and cloud API calls are allowed. Trade data and secrets are forced-local; research data may use cloud.

Reading the Key Metrics

  • Service status color — Green = healthy (latency < 100ms), yellow = degraded (100ms-1s or partial rate limiting), red = down (unavailable). If Freqtrade turns red, trade execution will be interrupted.
  • Provider failure rate — Percentage of AI service calls that failed. Consistently above 5% means you should switch routing strategies or check for expired API keys.
  • Inference queue backlog — A growing queue of pending jobs suggests the current Provider lacks throughput. Consider adding a fallback Provider or upgrading the model instance.
  • Routing strategy — failover (auto-switch to fallback on failure), round-robin (distribute load), local-only (no cloud calls), cost-opt (cheapest available Provider first).

Common Actions, Step by Step

  1. Open Service Management and scan the status grid — confirm every dot is green.
  2. If a service is yellow or red, click its "Test" button to see if it is a transient glitch.
  3. Move to the Provider overview area and check each Provider's failure rate and latency. If OpenAI is consistently slow, consider rerouting tasks to Anthropic.
  4. In the task routing matrix, adjust strategy — e.g. switch Generation from failover to cost-opt to reduce spend.
  5. Review the inference queue. Cancel long-waiting jobs or promote urgent ones.
  6. Check the privacy policy matrix to ensure sensitive data (trade data, secrets) is forced to local inference only.

Behind the Scenes

Service status data is fetched from backend endpoints /api/system/status and /api/system/dependencies. Freqtrade status is polled via its REST API; Redis and PostgreSQL are checked through the backend's connection pool. AI Provider data is fetched in parallel via APIAIProviders — Provider list, model runtime, routing rules, and privacy policy each travel on independent requests. The inference queue is managed by APIInference; job cancellation hits /api/inference/jobs/<id>/cancel. All data refreshes are triggered manually by the refresh button at the top of the page.
