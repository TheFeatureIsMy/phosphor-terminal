Backtest & Simulation

Page · Strategy · 5 min

Validate your strategy against historical data before committing real capital, or observe its real-time behavior in a paper trading environment.

TL;DR

Backtest and Simulation are the two stages of strategy validation. The BacktestLab page uses a two-column layout: a Run Rail (240pt) on the left lists historical backtest runs, and a scrollable narrative flow panel on the right contains 9 progressively unlocked sections. Backtests progress through five phases: idle → configuring → running → completed → failed. Sections 3-8 (Summary, Curve, TradeList, Compare, Risk, Promotion) only appear after completion. The page supports compare mode (check up to 3 runs in the Run Rail), a risk panel with 5 rules (Max Drawdown, Profit Factor, Trades, Win Rate, Sharpe), and two-level failure clustering (run-internal and strategy-level). The Promotion Panel summarizes 11 readiness gates, with a CTA that navigates to the Live Readiness page (it does not start live trading from this page).

Who uses it & when

Every strategy author. Backtesting is mandatory before going live — run it after every DSL or canvas change. The BacktestLab page (from the sidebar) offers the full backtest engine with Run Rail history and the narrative flow panel. Paper trading (simulation) is the final step after satisfactory backtest results, used before going live.

Page Anatomy

  1. ① Run Rail — The left 240pt rail lists historical backtest runs for this strategy. Each entry shows a run number, date, and return percentage (green for profit, red for loss). Each entry has a checkbox; select up to 3 to enter compare mode. The "Compare (N/3)" badge at the top shows the current selection count.
  2. ② Narrative flow panel — The right scrollable area contains 9 sections in sequence. Sections 1 (ConfigPanel) and 2 (StatusPanel) are visible during execution; sections 3-8 unlock only after the backtest completes (completed phase); section 9 (DataSourceFooter) is always visible.

Section Reference

  1. ConfigPanel — Backtest configuration. Set the date range (YYYY-MM-DD format), initial capital (USDT), and trading pairs (comma-separated). The strategy version is auto-selected from the current strategy. Click "Start Backtest" to submit. Always visible.
  2. StatusPanel — Execution progress panel. Shows the phase indicator (configuring → running) and elapsed time during backtest execution. Automatically transitions to results display on completion.
  3. SummaryPanel — 8 core KPI metric cards: Total Return, Sharpe Ratio, Max Drawdown, Win Rate, Profit Factor, Total Trades, Initial Capital, and Trading Pairs. Each metric is color-coded for quick assessment. Visible only in the completed phase.
  4. CurvePanel — Equity curve chart showing capital trends over the backtest period. Visible only in the completed phase.
  5. TradeListPanel — Trade-by-trade breakdown table listing time, side, price, quantity, and P&L for each trade. Visible only in the completed phase.
  6. ComparePanel — Compare panel. Expands when 2-3 runs are checked in the Run Rail, displaying a comparison matrix (side-by-side metrics) and an equity curve overlay chart. Visible only when completed and ≥2 runs are selected.
  7. RiskPanel — Risk rule panel. Checks 5 rules: Max Drawdown threshold, Profit Factor minimum, minimum trades, Win Rate minimum, and Sharpe Ratio minimum. Each rule shows pass / warn / fail. Visible only in the completed phase.
  8. PromotionPanel — Promotion gate panel. Summarizes 11 readiness checks (strategy version, backtest passed, risk compliance, etc.) with pass/fail status. The CTA button navigates to the Live Readiness page — it does NOT start live trading from this page, it serves as a readiness entry point only.
  9. DataSourceFooter — Data source info bar. Shows Freqtrade engine version, OHLCV data source, backtest execution time, and DSL hash. Always visible.

Reading the Key Metrics

  • Total Return — The percentage profit over the backtest period. Green for positive, red for negative. Always evaluate alongside Max Drawdown: 40% return with 5% drawdown is much better than 40% with 25%.
  • Sharpe Ratio — Risk-adjusted return. >=1.5 is excellent (green), 1.0-1.5 is acceptable (yellow), <1.0 is low. It tells you how much excess return you earn per unit of volatility.
  • Max Drawdown — The largest peak-to-trough decline during the backtest. Always shown in red. For crypto strategies, 20-40% drawdowns are common but depend on the strategy type.
  • Win Rate — The proportion of profitable trades. >=50% is green. But higher isn't always better — a high win rate with a low profit factor may underperform a lower win rate with a high profit factor.
  • Profit Factor — Gross profit / gross loss. >=1.5 is green. If the profit factor is high, the strategy can still be profitable even with a 40% win rate.

Common Actions, Step by Step

  1. Run a backtest — Open the BacktestLab page from the sidebar. Make sure the strategy has a saved version. Fill in the date range, initial capital, and trading pairs in ConfigPanel. Click "Start Backtest".
  2. Monitor progress — The backtest runs asynchronously. StatusPanel shows the current phase and elapsed time. The frontend polls every 3 seconds for status updates.
  3. Review results — After completion, SummaryPanel (metric cards), CurvePanel (equity curve), and TradeListPanel (trade list) appear automatically. Check the Sharpe Ratio and Max Drawdown to decide if the strategy needs optimization.
  4. Compare multiple runs — Check 2-3 historical runs in the Run Rail. The ComparePanel expands automatically with a side-by-side metrics matrix and an equity curve overlay.
  5. Check risk compliance — The RiskPanel evaluates 5 rules with pass/warn/fail indicators. Red-flagged rules require optimization before the next backtest.
  6. Check readiness — The PromotionPanel summarizes 11 readiness gates. Click the CTA to navigate to the Live Readiness page for full details (live trading is not started from this page).
  7. Review data sources — The DataSourceFooter displays the engine, data source, and execution info used for the backtest, useful for audit and reproducibility.

Failure Clustering

Backtest failures are diagnosed at two levels: Run-level failures — data source unavailable, engine crash, timeout; these can be retried by restarting. Strategy-level failures — DSL syntax error, strategy logic exception, indicator calculation error; these require strategy edits before resubmission. The StatusPanel explicitly labels the failure level to help quickly identify the root cause.

Paper Trading

Paper trading monitoring runs on a separate page. The strategy receives live market data and generates signals in a simulated environment without placing real orders. Open the Dryrun Monitor page from the sidebar to view active simulation runs. Each run card shows mode, status, start time, and the latest signal.

Behind the Scenes

Backtesting runs asynchronously via the Command Bus. When you click "Start Backtest", the frontend calls api.startBacktest() and receives a commandId. It then polls api.backtestStatus(commandId:) every 3 seconds. The page state machine transitions through idle → configuring → running → completed / failed; sections 3-8 unlock based on the state machine phase. In compare mode, the frontend collects the selected runId list and calls api.compareBacktests(runIds:) to fetch comparison matrix data. Risk rules are computed locally on the frontend by comparing KPIs from backtest results against preset thresholds. Paper trading runs are fetched via APIStrategyRuns.listRuns(mode: "dryrun").
