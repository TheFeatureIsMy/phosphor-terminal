Backtest & Simulation

Page · Strategy · 5 min

Validate your strategy against historical data before committing real capital, or observe its real-time behavior in a paper trading environment.

TL;DR

The BacktestLab page uses a three-column layout: the left Run Rail (240pt) lists historical runs and supports compare checkboxes, the center column holds a Backtest/Dryrun tab switch + a scrollable content panel (config → status/summary → equity curve+drawdown → trade list+failure clustering → compare), and the right Context Rail (280pt) always displays strategy metadata, risk warnings, and the promotion readiness gate. The old NewRunSheet modal and 9-section narrative flow have been removed — all operations are now inline in the main layout.

Who uses it & when

Every strategy author. Backtesting is mandatory before going live — run it after every DSL or canvas change. The BacktestLab page (from the sidebar) offers the full backtest engine. Paper trading (simulation) is the final step after satisfactory backtest results, used before going live.

Page Anatomy

  1. ① Run Rail (left) — A 240pt-wide vertical list of historical runs. Each entry shows the run number, date, and return percentage (green for profit, red for loss), with a checkbox on the right. Select up to 3 to enter compare mode. The "New Run" button at the bottom starts a new backtest or dryrun — no modal form needed.
  2. ② Center column — A Backtest / Dryrun tab bar at the top. Below is a scrollable content area: ConfigPanel is always visible; after completion, StatusSummaryBlock (8 KPIs), EquityCurveBlock (equity curve + drawdown), and TradeListBlock (trades + failure clustering) appear. When 2+ runs are checked, CompareBlock automatically expands with a comparison matrix and equity overlay.
  3. ③ Context Rail (right) — A 280pt-wide context inspector, always visible. Contains StrategyMetaPanel (name, version, data source), RiskWarningsPanel (5 rule checks, visible after completion), and PromotionGatePanel (11 readiness checks, CTA navigates to Live Readiness).

Section Reference

  1. Run Rail — The left rail lists all historical backtest and dryrun runs for the current strategy. Each entry displays the run number, date, return percentage, and status badge. Checkboxes enable compare mode (max 3). Click "New Run" to configure a new run inline in the ConfigPanel.
  2. ConfigPanel — The configuration panel at the top of the center column. Select the strategy version, trading pairs (comma-separated), date range (YYYY-MM-DD), initial capital (USDT), fee, and slippage. Click "Start" to submit. Always visible.
  3. StatusSummaryBlock — Execution status and summary panel. During execution, shows a phase indicator (configuring → running) and elapsed time. On completion, switches to 8 core KPI metric cards: Total Return, Sharpe Ratio, Max Drawdown, Win Rate, Profit Factor, Total Trades, Initial Capital, and Trading Pairs. Each metric is color-coded. Visible in the completed or failed phase.
  4. EquityCurveBlock — Equity curve chart with drawdown overlay. Shows capital trends over the backtest period with drawdown regions highlighted. Visible only in the completed phase.
  5. TradeListBlock — Trade-by-trade breakdown table listing time, side, price, quantity, and P&L for each trade. Also shows run-level and strategy-level failure clustering. Visible only in the completed phase.
  6. CompareBlock — Compare panel that expands when 2-3 runs are checked in the Run Rail. Displays a comparison matrix (side-by-side metrics) and an equity curve overlay chart. Visible only when completed and ≥2 runs are selected.
  7. StrategyMetaPanel (right rail) — Always-visible strategy metadata showing the current strategy name, DSL version, data source (Freqtrade engine, OHLCV source), execution time, and DSL hash.
  8. RiskWarningsPanel (right rail) — Risk rule panel checking 5 rules: Max Drawdown threshold, Profit Factor minimum, minimum trades, Win Rate minimum, and Sharpe Ratio minimum. Each rule shows pass / warn / fail. Visible after backtest completion.
  9. PromotionGatePanel (right rail) — Promotion readiness gate summarizing 11 readiness checks (strategy version, backtest passed, risk compliance, etc.) with pass/fail status. The CTA button navigates to the Live Readiness page — it does NOT start live trading from this page.

Reading the Key Metrics

  • Total Return — The percentage profit over the backtest period. Green for positive, red for negative. Always evaluate alongside Max Drawdown: 40% return with 5% drawdown is much better than 40% with 25%.
  • Sharpe Ratio — Risk-adjusted return. >=1.5 is excellent (green), 1.0-1.5 is acceptable (yellow), <1.0 is low. It tells you how much excess return you earn per unit of volatility.
  • Max Drawdown — The largest peak-to-trough decline during the backtest. Always shown in red. For crypto strategies, 20-40% drawdowns are common but depend on the strategy type.
  • Win Rate — The proportion of profitable trades. >=50% is green. But higher isn't always better — a high win rate with a low profit factor may underperform a lower win rate with a high profit factor.
  • Profit Factor — Gross profit / gross loss. >=1.5 is green. If the profit factor is high, the strategy can still be profitable even with a 40% win rate.

Common Actions, Step by Step

  1. Run a backtest — Open the BacktestLab page from the sidebar. Make sure the strategy has a saved version. Fill in the configuration in ConfigPanel and click "Start". A new run entry appears in the Run Rail with a configuring status.
  2. Monitor progress — The backtest runs asynchronously. StatusSummaryBlock shows the current phase and elapsed time. The frontend polls every 3 seconds for status updates.
  3. Review results — After completion, the center column automatically expands with StatusSummaryBlock (metric cards), EquityCurveBlock (equity curve), and TradeListBlock (trade list). Check the Sharpe Ratio and Max Drawdown to decide if the strategy needs optimization.
  4. Compare multiple runs — Check 2-3 historical runs in the Run Rail. The CompareBlock expands automatically with a side-by-side metrics matrix and an equity curve overlay.
  5. Check risk compliance — The right rail RiskWarningsPanel evaluates 5 rules with pass/warn/fail indicators. Red-flagged rules require optimization before the next backtest.
  6. Check readiness — The right rail PromotionGatePanel summarizes 11 readiness gates. Click the CTA to navigate to the Live Readiness page for full details (live trading is not started from this page).
  7. Review data sources — The right rail StrategyMetaPanel displays the engine, data source, and execution info used for the backtest, useful for audit and reproducibility.
  8. Switch to paper trading — Click the "Dryrun" tab at the top of the center column to switch to paper trading mode. Configure a dryrun the same way as a backtest; the Run Rail shows its running status.

Failure Clustering

Backtest failures are diagnosed at two levels: Run-level failures — data source unavailable, engine crash, timeout; these can be retried by restarting. Strategy-level failures — DSL syntax error, strategy logic exception, indicator calculation error; these require strategy edits before resubmission. StatusSummaryBlock and TradeListBlock explicitly label the failure level to help quickly identify the root cause.

Paper Trading

Paper trading is integrated into the BacktestLab page. Click the "Dryrun" tab at the top of the center column to switch to paper trading mode. The strategy receives live market data and generates signals in a simulated environment without placing real orders. The Run Rail shows active dryrun entries with mode, status, start time, and the latest signal.

Behind the Scenes

Backtesting runs asynchronously via the Command Bus. When you click "Start", the frontend calls api.startBacktest() and receives a commandId. It then polls api.backtestStatus(commandId:) every 3 seconds. The page state machine transitions through idle → configuring → running → completed / failed. In compare mode, the frontend collects the selected runId list and calls api.compareBacktests(runIds:) to fetch comparison matrix data. Risk rules are computed locally on the frontend by comparing KPIs from backtest results against preset thresholds. Paper trading runs are fetched via APIStrategyRuns.listRuns(mode: "dryrun").
