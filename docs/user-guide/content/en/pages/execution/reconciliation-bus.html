Reconciliation Bus

Page · Execution · 5 min

The Reconciliation Bus ensures that the position data you see in AlphaLoop matches the exchange's true state — every discrepancy is captured and flagged.

TL;DR

The Reconciliation Bus is AlphaLoop's "data consistency police." It periodically compares local database positions against the Freqtrade/exchange real state, finds discrepancies (local-only, remote-only, amount mismatches), and presents them as a Command Bus timeline and reconciliation run history. A "Refresh Exchange State" button at the top lets you trigger a full reconciliation cycle manually.

Who uses it & when

System administrators and advanced traders use this page when they suspect order state is out of sync (e.g., the local UI shows an order as filled but the exchange still has it open), after running backtests or dry-runs to verify data consistency, or when investigating "ghost position" problems. Regular users do not need to visit this page frequently — reconciliation runs automatically in the background.

Page Anatomy

  1. ① Header — Page title with subtitle "Command Bus · State Lease · Exchange Finality". The "Refresh Exchange State" button triggers a manual reconciliation cycle.
  2. ② Reconciliation Status — Overall health: healthy / reconciliating / degraded (discrepancies found) / error.
  3. ③ Command Bus Timeline — Last 20 command executions showing command type, ID, and status (completed / pending / failed / timeout). Commands include start_dryrun, place_order, stop_session, etc.
  4. ④ Reconciliation Records — Last 10 reconciliation runs, each showing the discrepancy count. 0 discrepancies is green (fully consistent), greater than 0 is orange (needs investigation). A yellow triangle indicates additional reason_codes.
  5. ⑤ Status Colors — Green=completed/success / Yellow=pending/running / Red=failed/error / Orange=timeout.

Reading the Key Metrics

  • Discrepancies — The number of mismatches found between local and exchange state during a reconciliation run. Zero is the ideal. Any non-zero value should be investigated. Common types: local_only (position exists locally but not on the exchange), remote_only (position exists on the exchange but not in local records), amount_mismatch (quantities do not match).
  • Reconciliation Status — healthy means the last run had zero discrepancies; reconciliating means a run is in progress; degraded means discrepancies were detected but the system continues running; error means the reconciliation process itself failed.
  • Command Bus Status — completed means the command executed successfully; pending/running means it is still being processed; failed means execution returned an error; timeout means the command exceeded its time limit (usually indicates a slow exchange response).

Common Actions, Step by Step

  1. Refresh exchange state — Click the "Refresh Exchange State" button in the top right. The system pulls the latest positions and orders from the Freqtrade REST API, compares them against local records, and updates the discrepancy list.
  2. View reconciliation details — Click a reconciliation run row to expand its discrepancy details (currently shown inline; a full drill-down view is in development).
  3. Investigate discrepancies — When discrepancies > 0, check the type: local_only usually means the local record was not synced to the exchange (possibly a network issue); remote_only means an exchange order was not written back locally (possibly a lost callback); amount_mismatch requires checking for partial fills that were not updated.

Behind the Scenes

The Reconciliation Bus is driven by ReconciliationService (reconciliation_service.py). The reconciliation flow has three steps: first, start_reconciliation creates a reconciliation event and marks the FreqtradeRun as "reconciliating"; second, execute compares local ExecutionPosition records against the positions returned by the Freqtrade REST API, checking each symbol for existence and amount consistency; third, complete marks the reconciliation as finished and restores the FreqtradeRun status to "running". Every step is written to the execution ledger as PULSEDESK_RECONCILIATION_STARTED / PULSEDESK_RECONCILIATION_COMPLETED events. The BFF endpoints /api/reconciliation/bus and /api/reconciliation/runs serve the frontend, querying the last 20 Command Bus entries and the last 50 reconciliation events.
