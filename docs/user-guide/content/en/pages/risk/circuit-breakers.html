Circuit Breakers

Page · Risk · 5 min

Circuit breakers are the last line of defense — when daily loss, consecutive losses, or volatility exceeds their thresholds, the system automatically halts all trading to protect the account from a single market event.

TL;DR

The Circuit Breakers page displays every breaker trip as a timeline. Filter chips at the top let you narrow by type: All, Emergency Stop, Kill Switch, Loss Lock, Manual Close, and Safe Mode. Each timeline entry shows the breaker type badge, relative timestamp, reason codes, and associated command/reconciliation IDs. When no breakers have ever been triggered, the page shows a pulsing green shield icon with "ALL CLEAR".

Who uses it & when

The Circuit Breakers page is primarily for post-incident review. When the system suddenly stops trading, first check the Execution Center for the current state, then come here to understand why. A quick scan at the end of each trading day is also good practice. If you're running a new strategy in live-small mode, check this page every few hours.

Page Anatomy

  1. 1. Filter Chips — Filter breaker records by type: All, Emergency Stop, Kill Switch, Loss Lock (daily + weekly combined), Manual Close, Safe Mode. Active filter is highlighted in the type's color.
  2. 2. Breaker Record (Daily Loss Lock) — Left-side timeline dot colored by breaker type. Card shows type badge, relative timestamp (e.g., "2h ago"), reason code chips, and associated command ID.
  3. 3. Breaker Record (Emergency Stop) — Red for manual emergency stop. Shows associated CMD (command) and RECON (reconciliation) IDs for audit trail.
  4. 4. All Clear State — When no breakers have ever been triggered, a pulsing green shield icon with "ALL CLEAR" text replaces the timeline.

Reading the Key Metrics

  • Breaker Type — Emergency Stop (manual, red), Kill Switch (automatic protection, red), Daily/Weekly Loss Lock (loss limit reached, orange-red), Manual Close (yellow), Safe Mode (purple). The type determines the recovery procedure after the breaker trips.
  • Trigger Time — Displayed as a relative timestamp ("just now", "5m ago", "2h ago", "3d ago"). More recent triggers need more urgent attention to the recovery state.
  • Reason Codes — The specific trigger cause, e.g., daily_loss_limit_reached (daily loss limit hit), manual_trigger (manual initiation), consecutive_loss_limit (consecutive loss limit exceeded).
  • Related CMD / RECON IDs — When a breaker was triggered by a specific command or is related to a reconciliation event, the associated IDs appear here for audit trail purposes.

Common Actions, Step by Step

  1. View all breaker history — Open the Circuit Breakers page. Records appear in reverse chronological order, newest first.
  2. Filter by type — Click a filter chip to show only that breaker type. Click the same chip again to clear the filter.
  3. Investigate a trigger — Examine the reason codes on each record. Common causes: daily_loss_limit_reached, consecutive_loss_limit, manual_trigger.
  4. Resume trading — This page is read-only for historical records. To resume trading after a breaker, go to the Risk Center to unlock, then the Execution Center to restart strategy sessions manually.

Behind the Scenes

Circuit breaker data comes from the /api/risk/circuit-breakers endpoint. The backend queries the circuit_breaker_events table in PostgreSQL for events of types emergency_stop, daily_loss_lock, weekly_loss_lock, kill_switch, and manual_force_close, ordered by creation time descending. Unresolved breaker events (resolved = false) set the overall state to "tripped"; all resolved yields "healthy". If the database query fails, the endpoint falls back to mock data. Breaker events are written by the risk logic in other backend services (e.g., the daily trading loop's circuit breaker check).
