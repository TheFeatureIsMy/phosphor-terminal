三种架构方案对比

目标：80-200+ 节点大规模画布，ProofAlpha 暗黑风格，ComfyUI + Node-RED 融合交互

A

渐进增强 Incremental Enhancement

保持现有三栏架构不动，逐项修复已知问题

Pros

  • 风险低，每步可独立发布
  • 改动范围可控
  • 现有代码逻辑不变

Cons

  • 架构债累积（ForEach 虚拟化是补丁）
  • 三栏布局在大画布下显得拥挤
  • 交互流不连贯（面板→画布→配置切换不自然）
B

纯 Canvas 2D 渲染 Custom Canvas Engine

放弃 SwiftUI View 节点，全部用 Canvas 2D 绘制 + 手势命中测试

Pros

  • 极致性能，200+ 节点流畅 60fps
  • 完全控制渲染管线
  • 视口裁剪天然支持

Cons

  • 无法使用 SwiftUI 原生控件（Slider/Toggle/Picker）
  • 需自建交互系统（焦点、键盘、可访问性）
  • 开发周期 3-4 倍于其他方案
  • 与 ProofAlpha 设计系统的 SwiftUI 组件割裂
C

混合重构 Hybrid Redesign ⭐ 推荐

保留 SwiftUI NodeView + 视口裁剪，重构面板交互流，深化连线系统

Pros

  • 原生控件可用（Slider/Toggle/Picker 在节点内）
  • 与 ProofAlpha 设计系统完全兼容
  • 手势和可访问性天然支持
  • 可增量交付，每层独立验证

Cons

  • 需实现视口裁剪层
  • 200 节点时需测试确保 60fps
  • 部分优化需改变数据流

方案 C 架构概览

重构后的画布架构 — 四层模型
第1层
CanvasBackground
点阵网格 + 扫描线叠加（已存在，性能 OK）
第2层
CanvasEdges（增强）
Bezier 曲线 + 端口精确定位 + 数据流方向粒子动画 + 选中高亮
第3层
ViewportCulledNodeLayer（新）
LazyVStack 替代 ForEach + viewport rect 计算 + 视口外节点不渲染（只保留占位）
第4层
Overlay 层
框选矩形 + 拖拽预览 + 对齐辅助线 + 连线即时类型校验指示器

方案 C 重点改造模块

🎯 画布核心
• ViewportCulling — 仅渲染可视区域内节点
• 精确端口连线 — 计算每个 port 的实际屏幕坐标
• 数据流方向动画 — 粒子沿边线流动
• Snap-to-grid + 对齐辅助线
🧩 节点系统
• 端口类型即时校验 — 连线时实时显示兼容性
• 节点体内控件增强 — Slider/Dropdown/Toggle 完整实现
• 折叠态摘要 — 折叠时显示关键指标预览
• 状态徽章 — 错误/警告/已连接 指示
📦 节点调色板
• 收藏夹 + 最近使用（本地持久化）
• 拖出到画布添加（Drag & Drop）
• 分类标签栏替代 DisclosureGroup
• 搜索高亮 + 模糊匹配
⚙ 配置面板
• 实时预览 — slider 变动即时反映到节点外观
• 输入校验 — 数值范围/必填字段即时反馈
• filePicker 完整实现
• 撤销重做 coalescing
